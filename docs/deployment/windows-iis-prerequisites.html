Windows Server / IIS Hosting Prerequisites

Target topology

Production runs as two Windows / IIS application servers behind a bank-owned F5 load balancer. Each app server hosts the application plus its own RabbitMQ broker (co-located, not shared). The nodes share one SQL Server, one Redis server, an SMTP relay and a NAS file share. Because there are two app nodes (N=2), every shared component (Redis backplane, NAS storage) must be common to both nodes.

Section A — On EACH IIS application server (install on every node)

#ComponentWhy it is needed
A1 Windows Server 2019 / 2022 with the IIS (Web Server) role and default features (Static Content, Default Document, HTTP Errors, Request Filtering) Base hosting platform for the application.
A2 .NET 9 Hosting Bundle (bundles the .NET Runtime + ASP.NET Core Runtime + ASP.NET Core Module v2 for IIS) Runs the app in-process inside IIS (w3wp). Install the Hosting Bundle, not the plain runtime — it registers the ANCM v2 IIS module. Reset IIS after install.
A3 IIS URL Rewrite Module (and ARR if IIS itself does any proxying) Commonly needed for HTTPS redirect / forwarded-header rules. Not required for plain in-process hosting but expected in most bank IIS setups.
A4 A Chromium-based browser — Google Chrome, Microsoft Edge, or any other Chromium-based browser (Brave, Vivaldi, a standalone Chromium build, etc.) The Reporting module renders PDFs with headless Chromium (PuppeteerSharp), which can drive any Chromium-based browser over the DevTools protocol — Chrome, Edge, or equivalent. Microsoft Edge is the easiest choice since it ships with Windows Server 2022 (no separate install). Point Reporting:ChromiumExecutablePath at the browser executable (e.g. msedge.exe or chrome.exe). Pre-installing any of these avoids the app auto-downloading ~150 MB of Chromium on first run.
A5 Erlang/OTP and RabbitMQ (both required, installed locally on each app server, not shared) Message broker for the MassTransit event bus (integration events, sagas, retries, dead-letter). Both are mandatory — RabbitMQ on Windows runs on the Erlang runtime, so install Erlang/OTP first, then RabbitMQ. Each node runs its own broker on 5672 (management UI 15672).

Section B — Backing infrastructure (separate servers / bank-provided)

#ComponentVersion / PortUsed for
B1 SQL Server 2019+ · 1433 Primary application database, Hangfire job store, Serilog log sink, OpenIddict store, and shared Data Protection keyring (required so antiforgery/token keys are common across both nodes).
B2 Redis 8+ · 6379 Distributed cache and the SignalR backplane. With two nodes the backplane is required so realtime messages reach clients regardless of which node they are connected to.
B3 SMTP relay (bank mail gateway) port 587 STARTTLS Outbound email (quotation notifications, meeting invitations) via MailKit. Allow outbound from both app servers to the relay.
B4 NAS / SMB file share (UNC path) SMB Shared storage for document uploads, generated report PDFs, temp and archive folders. With two nodes a shared UNC path is required so both serve the same files. App-pool identity needs read/write.
B5 SFTP server (AS400 reappraisal file drop) port 22 Source/processed directories for the monthly AS400 collateral-review files. Required for the reappraisal ingestion job, which reads the AS400 collateral-review files over SFTP in production.
B6 Active Directory / LDAP directory server LDAPS 636 Users sign in with their AD / bank domain accounts (Ldap:Enabled = true). Each app server needs outbound access to the directory server over LDAPS; the bind service account credentials go in app configuration.

Note: these backing services are dependencies the app connects to — they are normally installed on their own servers or provided as managed/bank infrastructure, not on the IIS app servers.
