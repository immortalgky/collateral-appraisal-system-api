Everything is configuration. Each activity has two knobs — Clock (Own clock vs Part of a window) and Anchor (Assignment vs Appointment date). The engine just reads it: an activity assigned to a window is governed by that window; otherwise its own clock runs from its anchor. The current "external = window, internal = per-activity, execution = appointment" is only the default seed, fully changeable.

Why "total consumption", not "this round"

If the screen showed only the current round, the right card would read 20h / 72h — On track even though the work has actually eaten 88h across 2 rounds. Showing cumulative keeps the signal honest and ungameable.

How a deadline is computed

1
Resolve the anchor. If the activity's policy anchor is Appointment date → use the assignment's latest non-cancelled appointment. Else → use AssignedAt. No appointment yet ⇒ deadline is "awaiting appointment" (not started).
2
Sum prior consumption. Add up the business-time of this activity's previous completed runs (from CompletedTasks), excluding supervisor-reassign rows. First pass ⇒ 0.
3
Compute the deadline. Due = anchor + (budget − cumulative) in business-time (skips weekends / holidays / lunch). If budget − cumulative ≤ 0 ⇒ already breached.
4
Display total consumption. The task shows cumulative / budget · round N. "Remaining" = budget − cumulative — goes red when over. The existing SLA monitor watches Due, so at-risk / breach alerts follow the cumulative automatically.
5
On reschedule. If the appointment moves, the anchor-bound deadlines are recomputed (execution task + vendor window). Other (assignment- anchored) activities are untouched.

Scenarios — External Appraisal Execution (budget 72h)

① Normal — appointment set, delivered on time

28 Jun — task assigned
Clock does not start (anchor = appointment). Status: Awaiting appointment
01 Jul 09:00 — appointment (on-site visit)
Clock starts. Due = 01 Jul + 72h = 04 Jul 17:00.
03 Jul 14:00 — submitted
Consumed ~45h of 72h → Met

② Future appointment — early assignment doesn't burn time

28 Jun — task assigned, appointment booked for 10 Jul
Days pass, but consumption stays 0h — the clock hasn't started. Status: Awaiting appointment
10 Jul — appointment
Now the 72h budget begins. The 12 idle days are correctly not counted against the appraiser.
This is the whole point of the appointment anchor — assigning early can't penalize the appraiser.

③ Rework — cumulative continues (does NOT refresh)

01 Jul — appointment, Round 1 starts
Budget 72h.
04 Jul — submitted (68h used), bank routes back (quality)
Round 1 consumed 68h of 72h.
05 Jul — Round 2 (rework) starts
Remaining = 72 − 68 = 4h. Due = now + 4h. The task shows 88h / 72h — Breached as soon as Round 2 passes 4h.
Contrast: if rework refreshed, Round 2 would show 20h / 72h — On track and hide that the work really took 88h. Cumulative prevents that.

④ Reschedule — deadlines recalculated

01 Jul appointment → Due 04 Jul 17:00
Initial.
02 Jul — appointment rescheduled to 08 Jul (approved)
Execution Due + vendor window are recomputed → new Due = 08 Jul + remaining budget. The appointmentDate variable updates; monitor re-evaluates.
Other activities
Assignment-anchored steps (initiation, check, approval…) are untouched by the reschedule.

External path — governed by ONE vendor window

On the external path the company is held to a single window (the whole vendor turnaround). The four ext-* tasks are just stages inside it — they show progress, not separate enforced clocks.

The window runs continuously from the appointment; each stage adds to the same 90h. Whoever holds the task (checker now) sees this one number — "90h / 144h" — not a private per-step clock.
Why one window: the bank's commitment is with the company, not with its internal hand-offs. So the company is measured on the total turnaround; the per-stage split is a drill-down for the scorecard, not four separate budgets that could each look "green" while the whole thing drags.

Rework: a routeback into the loop opens a new cycle but keeps adding to the same window total — so a re-checked book that drags the company past 144h shows breached, regardless of which stage caused it.
Decided: the window starts at the appointment (post-visit appraisal turnaround). The pre-visit admin step (ext-appraisal-assignment) is its own small clock and is not counted in the vendor window. The internal path keeps per-activity clocks (it's not window-governed) — int-appraisal-execution appointment-anchored, check / verification assignment-anchored.
