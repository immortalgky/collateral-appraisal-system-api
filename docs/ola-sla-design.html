1 · Concept — OLA vs SLA (ownership split)

SLA is the promise to the customer. OLA is the internal accountability behind it.

SLA

End-to-end customer commitment — request submitted → appraisal completed. One umbrella number per path, set independently (the business promise).

OLA

Internal / inter-party targets — the building blocks. The external company's turnaround is vendor OLA; the bank's own steps are internal OLA. They live within the SLA umbrella.

The key word is accountability. "This appraisal took 12 days" doesn't tell you who was slow. OLA splits the time by the party responsible for each chunk.

2 · What the targets vary by

OLA/SLA targets are configured per Loan Type × Appraisal Type:

  • Loan Type — the banking-segment value (Retail / IBG), reused as the SlaPolicy.LoanType dimension.
  • Appraisal Type — New / ReAppraisal / Progressive / PreAppraisal.

Stored as nullable wildcard rows: a fully-specified cell wins; a null acts as "applies to all". Existing seeded defaults (all-null) keep working as universal fallbacks — no data backfill needed. Assignment type is a deferred future axis.

3 · Activity ownership & scenarios

Each activity carries two independent attributes:

  • Owner — who is responsible for the time (drives display group + OLA attribution).
  • Scenario — which mutually-exclusive case it runs in.

An appraisal runs either the external-company case or the in-house case — never both:

Owner groupActivitiesScenario
Shared
bank, both cases		 appraisal-initiation-check, appraisal-initiation, int-pma-input, appraisal-assignment, int-appraisal-check, int-appraisal-verification, pending-meeting, pending-approval Both
External / Vendor
external company		 ext-appraisal-assignment, ext-appraisal-execution, ext-appraisal-check, ext-appraisal-verification External case
Bank appraisal step appraisal-book-verification (bank verifies the vendor's book) External case
int-appraisal-execution (in-house appraisal) In-house case
"External SLA" = vendor ext-* only. appraisal-book-verification is bank staff time — it runs only in the external case but is never counted against the vendor's OLA.
The mutually-exclusive pair: appraisal-book-verification (external case) and int-appraisal-execution (in-house case) are the bank's "first appraisal step" in each scenario — they never happen together. Meanwhile int-appraisal-check / int-appraisal-verification happen in both cases, so they're Shared.

4 · The config screen

Selector-driven (Loan Type × Appraisal Type), simplified to what's actually needed:

  • SLA umbrella — one end-to-end target field.
  • Per-activity hours — grouped by owner (Shared / External / Bank), each row a name + hours + Save, with override/inherited badges and a revert.
  • Group windows (Start → End + hours) — a shared OLA/SLA window over a span (e.g. the external company's total turnaround). Create / edit / delete (override → revert; default → global delete with confirmation).
Earlier iterations showed Σ-sum bars and "inner — not summed" coverage tags. Those were removed — per-activity hours plus configurable group windows are enough, without the sum machinery.

Gated by a new SLA_CONFIG_MANAGE permission, route admin/sla-config, menu node seeded; all idempotent on the next deploy (no backfill).

5 · How the engine assigns SLA

There are two independent clocks — and they behave oppositely.

ClockApplied toOn forward to the next step
Per-activity SLA
PendingTask.DueAt		 each activity's own deadline (its policy, or the workflow JSON timeout) restarts — a fresh clock from 0 when the next activity is assigned
Group / stage window
AppraisalAssignment.SLADueDate		 one deadline on the assignment, anchored to the Start activity continues — one envelope across the whole span, accumulates, includes idle gaps
Worked example. Group window ext-execution → ext-verification = 144h. ext-staff takes 7h, 2h handoff gap, then ext-check takes 5h:
  • Group window: 7 → continues through the gap → 14h / 144 (it never resets until the End).
  • ext-execution own SLA: 7h, done. ext-check own SLA: fresh 0 → 5h.
They're nested: the group is the total budget; each activity has its own restarting sub-budget. A group catches "every step was on time but the whole thing still took too long" (including waiting between steps).
Today the group deadline is Start + duration; EndActivityKey and MiddleActivityKeys are not read at runtime — they're documentation only.

6 · How "delivered" is detected (SubmittedAt)

AppraisalAssignment.SubmittedAt is not derived from the group's End activity. It's stamped by a hardcoded workflow transition edge in MarkUnderReview():

var isFirstSubmission = AssignmentStatus == InProgress && SubmittedAt is null;
AssignmentStatus = UnderReview;
if (isFirstSubmission) SubmittedAt = DateTime.Now;
  • External case: transition ext-appraisal-verification → appraisal-book-verification (vendor hands the book to the bank).
  • In-house case: transition int-appraisal-execution → int-appraisal-check.
  • First submission only — not re-stamped on rework/routeback.
So the vendor's "end" is already captured by SubmittedAt — which is why EndActivityKey on the group config is redundant. A cleaner alternative is to detect the window's end from the configured End activity's completion in CompletedTasks, which makes Start/End genuinely meaningful and is consistent with how the OLA reports already measure segments.

7 · Clock-state & responsibility (forward + replay)

Governing principle: a clock attaches to responsibility, not to activities.

It runs while a party holds the baton, stops when the baton leaves, and pauses when they're blocked on someone else. Replay (routeback) = the baton comes back = a NEW accountable span — it never rewinds the closed one. The appraisal workflow has ~20 routeback edges, so this matters.

Three clock scopes, three replay behaviors

ScopeOn replayNotes
Per-activity (leg)restart every execution each CompletedTask row is one leg; Movement="B" tags a rework leg; report first-pass vs rework separately
Per-party OLA (cycle)per engagement cycle; accumulate within; attribute across vendor already has this (ExternalEngagementCycle + TotalExternalBusinessMinutes)
End-to-end SLAcontinuous across all replays bank owns rework time; pause only for genuine "waiting on customer / third party"

Fault attribution

Whether a party's rework cycle counts against them depends on who caused the replay:

  • Reviewer quality-rejection (vendor's fault) → the rework time counts against the vendor.
  • Bank changed requirements / "request more info" → excluded from the vendor's OLA.
Gap: the routeback reason is transient (workflow variables + fire-and-forget events, not on the task log) and the responsible party / fault owner is recorded nowhere. Fair cross-cycle measurement needs this persisted.

What we'd build on

Already exists (reuse)
  • workflow.CompletedTasks — leg-grained log; rework ⇒ multiple rows per (CorrelationId, ActivityId)
  • Movement F / B / C on every leg
  • IBusinessTimeCalculator — business-time (holidays/lunch)
  • ExternalEngagementCycle — per-cycle accumulation template (vendor)
  • OlaTimingService — group-by + first-round reduction
Missing (to build)
  • per-leg business-minutes not persisted
  • bank-side cycle accumulation (only vendor exists)
  • routeback reason/owner durability (fault attribution)
  • explicit end-to-end "pause" state

8 · Detecting vendor SLA

Vendor SLA = actual turnaround (business time) vs target, from timestamps the assignment already records:

PieceField
clock startAppraisalAssignment.AssignedAt
clock endAppraisalAssignment.SubmittedAt
elapsedIBusinessTimeCalculator.GetBusinessMinutesBetweenAsync(...)
multi-roundTotalExternalBusinessMinutes, Cycles, SubmissionCount
targetgroup/stage policy hours or delivery thresholds

Two detection flavors already half-exist:

  • Graded (live) — DetectDeliveryTimeQuery maps turnaround to a 1–5 rating via EvaluationCriteriaConfigs thresholds → feeds company scoring.
  • Binary met/missed (dormant) — vw_AssignmentList derives SLAStatus from SubmittedAt ≤ SLADueDate, but nothing reads that view yet.
Pick one source of truth for "how long the vendor should take" — the group/stage policy and the delivery thresholds both encode it. Otherwise a vendor can be "MetSLA" yet scored 2/5.

9 · Implementation plan — Vendor + Bank party-SLA

Decisions: full Vendor + Bank split; measure raw first (defer fault attribution).

Approach: a single on-read evaluator over CompletedTasks — no new tables, no new event handlers.

  1. PartySlaEvaluator (Reporting module, beside OlaTimingService) — load legs for the appraisal, exclude "Reassigned" audit rows, classify owner (ext-* → Vendor, else Bank).
  2. Derive cycles — order legs by AssignedAt, group consecutive same-owner legs; ownership flip = cycle boundary.
  3. Per cycle — businessTime(firstLeg.AssignedAt → lastLeg.CompletedAt) (includes within-cycle gaps, excludes the other party's time); tag IsRework.
  4. Compare each party's elapsed (per-cycle + cumulative) to the configured group/stage target → met/missed.
  5. Surface — GET /appraisals/{id}/party-sla with the per-party cycle breakdown.

Deferred: routeback reason/owner persistence (fault attribution), explicit end-to-end pause states, live at-risk monitoring on party cycles, any persisted bank-cycle table.
